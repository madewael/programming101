Programming 101

Programming101 presents a collection of articles that each explain a computer science topic by building an executable toy version. Hence, the focus is always understandability over completeness.

The first part tackles programming languages. Foremost because it covers by background and education, but also because it forms the basics for further articles. When we build a relational database in Part II we will want to use SQL to query it. Well, SQL is a domain specific programming language (DSL) which will need a parser and an interpreter...

Part I : Programming Languages

Intro

In Part I: Programming Languages we will be building an extremely simple programming language from the ground up. The goal is to turn a sequence of characters, the source code, into a useful computation.

A first step is to turn source code into a sequence of tokens, much like humans can detect sentences in a text and words in a sentence. This is called lexing and is done by a lexer (see lexing).

A second step is to turn the sequence of tokens into an abstract syntax tree (AST), much like humans expect a question (sentence) to be followed by an answer (sentence) or expect a sentence to at least contain a subject (word or words) and a predicate (word or words). This is called parsing and is done by a parser (see parsing).

The third step depends on the kind of language. From an execution point of view, there are two kinds of languages: interpreted languages and compiled languages. An interpreted language uses an interpreter to directly perform the computation (see interpretation). A compiled language relies on a compiler to create new (executable) code (see compiling). Both the interpreter and the compiler need to get their hands on an abstract syntax tree (AST).

The programming language we will be building is very simple. It has only two (first-class) data types: strings and integers. Every program consists of a bunch (zero or more) function-definitions, followed by a single expression, and ends with a single ;. This is shown in The Grammar of our toy programming language. Everything between " is to interpreted as literal text. *, ?, and | are to be interpreted as zero or more times, zero or one time, either the one or the other (or), respectively. Finally, [ - ] denotes a range, e.g., any small letter can be written as ["a"-"z"]. All other constructs are called non-terminals and need to be looked up in the grammar. block, for instance, is defined as "{" statement* "}": zero or more statements surrounded by curly braces, a statement can be either of 5 options, ...

Lexing

In computer science, a lexer is a program that converts a sequence of characters into a sequence of tokens. Each token is a part of the original sequence to which the lexer assigns a meaning. For our toy programming language we will want to start with writing a lexer. This lexer will convert a sequence of characters into symbols, keywords, literals, and identifiers. At the same time, our lexer will ignore all white space.

The main data structure we will be using in lexing (and in parsing) is the Queue.

.

In our first iteration the lexer will not support symbols with a length larger than 1 (e.g., ==), or string literals. We add this functionality in a second pass. If we consider the grammar with these simplifications we obtain the following symbols: ; , ( ) = { } + - * / % < > !. Any of these symbols is always a token because they are not supposed to appear any where else: not in keywords, not in integers, not in identifiers. The same holds for whitespaces \t, \n, and \r, but these are also ignored.

The general approach for this simple lexer is to consume all characters from the input sequence one by one, hence the use of a queue. We accumulate all these characters into the next token. When we encounter either a symbol, a whitespace, or the end of the file (EOF), we know we processed a complete token and we remember it. If we just encountered a symbol, we need to remember this symbol as well.

Extending the Lexer

You can continue reading here if you want the expand the functionality of the lexer a bit more. We will be adding extra keywords which requires almost no work. We will add support for symbols that consist of more than one character, which will take a bit more effort. And finally we will add support for string literals, which will prove to be the most difficult endeavour of the three. If you want to get your first programming language up, and more importantly, running as soon as possible, you should skip this section and continue reading about parsing.

Extra Keywords

By simply adding if and else as keywords to the lexer, we are now able to turn a program with an if-statement into a correct sequence of tokens, i.e., where if and else are recognised as keywords instead of identifiers.

Multi Character Symbols

When we would add % to the list of lexable symbols, the lexer would simply recognise this character as a symbol. Simply adding ==, ==, and == to the list of symbols, however, is not sufficient to make them lexable. This is because the lexer is to eager to turn characters into tokens. A first step in our lexer extension it to make the lexer a bit more cautious. Instead of eagerly consuming characters from the queue, we will start by peeking. When we encounter a whitespace, we consume it just like before. When we encounter a character that is known not to be a symbol (last branch of the if), we also consume it right away and push into the auxiliary token buffer. For the first and the last branch, nothing actually changed.

For symbols, however, we introduce a new dedicated lexer, a function whose only purpose is to interpret the current state of a queue as if it starts with a symbol (of unknown length). First note that all new symbols have existing symbols as prefixes. Here, this means that the first character of a all symbols are symbols as well. Here, in this language we can rely on this fact. If this is not the case, the lexer would become even more complex.

We only enter our dedicated lexer in the event of detecting a (prefix of) a symbol. Hence, it is safe to consume the first character in the queue, since it is known to be a symbol. If we add the next character to the first character and we still have a (known) symbol, we have found a two-character symbol. If not, our symbol was one of length one. To check this, we need to peek again. This technique, first using peek in order to consume conditionally, is called looking ahead. This can extended for symbols of any length by using a while instead of an if. Note that this simple implementation only works is all symbols have another symbol as prefix ...

String Literals

The final extension to our lexer is the support for string literals. A string literal starts and ends with ". We could add this as a symbol to our lexer, but actually it is only used to denote the start and end of a string-literal-lexer-token, and has further no meaning for the parser for instance. Furthermore, the hardest thing about lexing a string literal is that most invariants that hold when lexing non-string literals do not hold: whitespaces are not to be ignored, symbols are not supposed to to delimit any token, and keywords hold no meaning. Hence, we introduce a second dedicated lexer, just like we did for symbols, using the same look ahead technique. The dedicated lexer for string literals will prove to be as complex as the lexer for all other items.

This works fine for nice and simple strings, e.q., Hello World.. But lexing a strings such as Hamlet: "To be or not to be, ...", where " are actually part of the string itself. Many programming languages resort to the technique called escaping where an new symbol is introduced to denote that the next character(s) need to be lexed differently. The symbol we will be using is equally classic as the technique itself: \. Thus, whenever we encounter a \ in a string token, we know we should interpret the following differently. We introduced \ to be able to lex ", thus a \ followed by a " is interpreted as an double quote. This solves the issue for double quotes, but creates an issue for backslashes. The solution is to make backslashes also escaped: \\ is to be interpreted as a backslash. Finally, we can piggyback on the escaping to encode/interpreted special characters such as return, or illegal characters such as newline (many languages do not allow newlines are in strings). If we encounter a non-escapable character after \, the lexer will fail.

Do not confuse the \ symbol with the symbols we already know from earlier, which have an actual meaning in our programming language. What we observe here is that strings actually a sort of (embedded) domain specific language (hosted) in another programming language. The same holds for integers but we where able to hide this complexity behind the abstraction offered to use by JS in the form of the parseInt-function.

This finalizes our lexer, which turns a sequence of characters into a sequence of tokes: symbols, keywords, strings, integers or identifiers. This sequence is further used as input for the parser, which is discussed in the next chapter.

Parsing

The main job of a parser is to turn the sequence of tokens into a more structured representation which is called the abstract syntax tree (AST). While doing so, the parser verifies whether the input has a correct structure, i.e., checking if the input matches the grammar.

As stated above, the input of a parser is a sequence of tokens. The result should be an AST that represents the full program. The main function parse turns the sequence of tokens into a queue of tokens and delegates to the dedicated parser parseProgram. For now the dedicated parser parseProgram does not do much, except show the structure of all dedicated parsers. Each dedicated parser accepts a queue and returns a structure that matches one of the grammar rules. Here, we parse a program, hence we expect a bunch of function definitions (zero, one, or more) and a single expression. Each program ends with a ; but that is not of any meaning in an AST. Hence the name abstract syntax tree, not all concrete syntax elements need to be included, only those that still matter. In the concrete syntax the ; is needed to detect the end of the expression. What remains is to implement functions that can parse (lists) of function definitions, that can parse expressions, and that can parse a ;.

We updated our Queue to throw an error whenever peek or consume is called on an empty queue. The code can be seen here. Because the update is trivial, the new version is omitted from the main text.

A more important update is that of the dedicated parser parseProgram which now delegates its work to an auxiliary parser and two dedicated parsers: parseFunctionDefinitionList, parseExpression , and parseSemicolon, respectively. Then, it combines the results into a new AST node that represents the complete program. Note that the expected semicolon is effectively parsed, but it is ignored afterwards. Finally, we check if the queue is empty, if not, we have found an illegal program and we throw a parse error.

For now, an implementation for parseFunctionDefinitionList is omitted. Hence, only programs without any functions are accepted. This is, if parseExpression where to be implemented completely. We again limit ourselves here, and only accept literal integer expressions. To do so, parseExpression delegates its work to parseInteger. This is our first fully functional dedicated parser: a literal integer expression always consists of exactly one lexer token, and it has to be of type integer. If this is the case, we transform the lexer token in an AST node of type integer. The value we assign to this node is the value we obtain, from the token. Conceptually this is a big step, in practice almost nothing changes here. Finally, it is important to note that our (dedicated) parser throws a parse error. When you want (or need) to parse an integer and you find something else, it is useless for this (dedicated) parser to continue. We will see later how and when we can recover from parse errors.

The implementation for the dedicated parser parseSemicolon is if possible even more simple: we consume the next token and verify its type and value. Here, we expect it to the symbol ;. If this is not the case, we encountered a parse error, hence the throw. We return the token, however this is not strictly necessary. As you can imagine, this kind of dedicated parser will have to be written for each and every symbol and keyword. Hence, we will move the logic to the Token, which will simplify the parsers a bit.

We add two functions to our Tokens:is and ensure. is will be limited to checking whether a tokens is of the expected type and value, whereas ensure will also enforce the expected condition by throwing a parse error is the conditions are not met. Moreover, we make the expected value optional, which will prove to come in handy as well... In the remainder of this text we will omit explicit references to the simple dedicated parsers, i.e., the parsers for keywords and symbols.

We now focus on our function definitions that may or may not precede the expression. In order the parse a bunch of these definitions, we need to be able to at least parse one such function definition. Hence, we need the dedicated parser parseFunctionDefinition. It very much resembles the dedicated parser parseProgram, in that is a straightforward translation from the grammar rule to executable code. The actual parsing of the parameter list and the body is delegated to the dedicated parsers parseParamList and parseBlock, respectively.

In their current implementations, parseParamList and parseBlock accept anything between the first open and close brackets it encounters (round or curly), they ignore the actual structure and return a simple list of un-parsed tokens. Hence, the only way to make them fail is by having an open bracket that is never closed. In this case the queue will run out of tokens and throw a parse error.

The grammar, however, states that programs have a list of these function definitions. Hence, we could use parseFunctionDefinition to parse the first such definition and return it as an element is a list. If there is no function definition, the parse would fail because it expects to find the keyword function but will find an integer token. We therefore wrap our code in a try-catch-block to recover from an more or less expected parse error. When our parser reaches the catch block, we know the code does not start with a proper function definition, hence it should start parsing the expression, hence, the list of functions is empty.

When we run our current parser of some demo programs, we see that it works fine for programs that have exactly one function definition followed by a literal integer as expression. However, we expected our parser to also accept the programs it accepted before, i.e., the programs that consist of a single literal integer without any functions. We observe, however, that parsing this integer fails. Even though we know the program is correct, we still get the following error: Parse error: expected a integer but got a symbol. The parse error thus occurs during the execution of parseInteger, parseFunctionDefinition and parseFunctionDefinitionList worked fine.

So ... what went wrong? The problem with our dedicated parsers is that they manipulate the queue. In programs that do not start with a function definition, we do try to parse a function definition. When it fails, we return an empty list and continue to try and parse an integer literal. Trying to parse the non-existing function definition, however, consumed the first token from the queue: the integer. Because parseFunctionDefinition expected the keyword function but received an integer-token, it fails fast. While parseFunctionDefinitionList is made to recover from this expected parse error, the harm is already done: the queue was changed and the integer was consumed, and is never to be found again.

A possible solution to this problem is to anticipate the destruction of the queue in those cases where a parse error is to be expected. If we where to copy the contents of the queue before parsing, and restore the queue to its original state if a parser error would occur, all would be fine. This is realised by tryParse: it expects a (dedicated) parser and a queue and behaves exactly as if the parser would have been called with the queue directly, i.e., return an AST node or throw a parse error. When an AST node is returned, the queue will have changed. In the case a parse error is thrown, however, the queue will be seemingly untouched.

We can now update parseFunctionDefinitionList such that it accepts any number of function definitions, by trying to parse them in an infinite loop. When we exit the loop, we know the last function definition has been successfully parsed, and the queue is restored to the state right after this last function definition.

Now we know how we can tryout one dedicated parser and continue with another if the first one failed, we can create the parseStatement function. If we take a look at the grammar rule, we see that a statement does not really exist, but is rather one of five more specific kinds of statement. In an object oriented language Statement would be modeled as an abstract type. Hence, we can implements parseStatement by trying the dedicated parsers one by one. If one of these parsers is able to create an AST node, the parseStatement returns this node. This endeavour must of course result in an AST node, hence, if none of the parsers is able to produce an AST node, we must conclude that we encountered something that is not a statement, hence the parse error.

The dedicated parsers we need for parseStatement are: parseVariableDefinition, parseVariableAssignment, parseWhileStatement, parseReturnStatement , parseBlock. The last one already exists, but will be refined later on. The first one is ignored for now.

Now that parseStatement exists, it is possible to improve our implementation of parseBlock. Up until now parseBlock simply checks for { and consumes tokens until it finds a }. In practice, however, we are not interested in a sequence of tokens as body of the block, but rather want a sequence of statements. We can simply parse a {, then keep on parsing actual statements using parseStatement. As soon as this fails, we know we parsed the last available statement, and expect the next token to be the closing curly bracket (}). In order for this to work correctly, do not forget to use tryParse in the loop.

When implementing parseStatement above, we skipped parseVariableDefinition because it inhibits a peculiarity not found in the other grammar rules: an optional part. The optional part in a variable definition defines the initial value of this variable, it can be present in the form of an expression, or absent. Concretely, we realise this by trying to parse a = followed by an expression. When this fails, we know the optional part is must be empty. We hide this logic by abstracting it away in the function parseOptionally which behaves just like tryParse but returns false instead of throwing a parse error in the event of a failed parse.

We now update the parseParamList parse to use the same technique as parseBlock, i.e., keep on parsing while putting the results in a list. There is one big difference, however, and it is not () versus {}. Note that a list of statements is just a list of statements. The parameter names in a parameter list, however are separated by ,-symbols. We solve this by using one of the possible techniques, that is keeping a flag that remembers whether this is the first parameter or not. This is technically realised by wrapping the parseIdentifier parser.

Besides the kind of brackets, i.e., () versus {}; and the presence or absence of a separator, parseParamList and parseBlock look very much alike. Later on, when we will be parsing function calls, we will see that the argument list follows a similar pattern. Moreover, parseFunctionDefinitionList also looks very similar: it simply has no start and no stop symbol, neither does it require a separator. Hence, we apply some functional abstraction and introduce an auxiliary parser function parseSequence, which takes the start, stop and separator symbol, and most importantly, a dedicated parser. It returns a list with the resulting AST nodes. The function itself is the most clumsy looking one in the project hitherto, but using parseSequence in parseFunctionDefinitionList, parseBlock, and parseParamList results in rather clean code. Note that parseParamList requires one post-processing step, as we are interested in the values only and not the complete tokes, hence the use of map.

We now have a parser for the lexer defined before here.
